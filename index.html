Sistem Informasi Point of Sale (POS)

Dokumentasi Komprehensif Perancangan UML dan Implementasi Aplikasi POS Kedai Dafo
Oleh: Jonny Ficky Felyang (NoBP: 2310112610508)

Pendahuluan

📖 Latar Belakang Masalah

Mengapa Kedai Dafo membutuhkan digitalisasi operasional melalui sistem POS terintegrasi?

Pencatatan Transaksi Lambat

Proses kasir yang masih menggunakan kalkulator manual atau buku kas berpotensi menghambat antrean pelanggan pada jam sibuk.

Ketidakakuratan Stok

Tanpa sistem terkomputerisasi, pengurangan stok barang sering kali terlewat, menyebabkan kehabisan stok bahan tanpa pemberitahuan dini.

Monitoring Keuangan Lemah

Owner kesulitan merekap total pendapatan harian, laba bersih, serta tren menu terlaris secara akurat tanpa kalkulasi manual yang memakan waktu.

Solusi & Sasaran

🎯 Tujuan Pengembangan Sistem

Solusi digitalisasi "Smart Warung POS" dirancang untuk menyelesaikan titik kendala utama bisnis.

Sasaran Operasional POS

  • Efisiensi Kasir: Menyediakan halaman checkout cepat dengan penghitungan total belanja otomatis dan validasi kembalian.
  • Sinkronisasi Inventori Real-Time: Pengurangan stok otomatis setiap kali item terjual dan deteksi stok menipis.
  • Kemudahan Pelaporan Owner: Laporan transaksi harian otomatis dengan statistik omzet dan opsi ekspor PDF.
  • Multi-Role Authentication: Keamanan akses berbeda antara Kasir (hanya transaksi) dan Owner (hak kontrol penuh, stok, & laporan).

Nilai Tambah

Smart Warung POS mengubah proses administrasi tradisional menjadi berbasis sistem yang cepat, presisi, dan aman. Meminimalisir deviasi pencatatan keuangan dan memberikan visibilitas operasional penuh bagi owner dari perangkat mana pun.

Pemodelan Sistem

📊 Use Case Diagram

Memetakan fungsionalitas sistem yang tersedia bagi masing-masing aktor (Kasir dan Owner).

Analisis Fungsional

Aktor Kasir: Melakukan Login, input data checkout, mengelola transaksi kasir, dan mencetak struk transaksi penjualan.

Aktor Owner: Melakukan Login, mengelola item stok/barang dagangan, melihat ringkasan dashboard, dan meninjau laporan keuangan bulanan.

Hubungan Sistem: Use case utama dilindungi oleh relasi <<include>> login untuk memastikan setiap data transaksional tercatat atas nama kasir/user yang sah.

Struktur Data

🏗 Class Diagram

Representasi hubungan antarentitas database relasional SQLite dalam aplikasi POS.

Entitas Utama Database

User: Menyimpan username, hashed password, dan hak akses/role (Kasir/Owner).

Barang: Menyimpan kode_barang, nama, harga jual, dan jumlah sisa stok.

Transaksi: Menyimpan total nominal pembayaran, ID transaksi, status bayar, dan waktu.

DetailTransaksi: Menyimpan kaitan relasi multi-barang dalam satu transaksi (*Many-to-Many*).

Proses Bisnis

🔄 Activity Diagram: Login

Alur proses validasi kredensial pengguna oleh sistem kasir POS.

Deskripsi Alur

1. User memasukkan username dan password ke form login.

2. Controller memvalidasi username dan password ke database SQLite.

3. Jika data cocok, session diaktifkan dan dialihkan ke dashboard.

4. Jika gagal, memunculkan notifikasi error dan tetap di halaman login.

Proses Bisnis

🔄 Activity Diagram: Penjualan

Alur operasional kasir dalam memproses checkout belanjaan barang.

Deskripsi Alur

1. Kasir memindai barcode produk atau memilih item.

2. Sistem menghitung subtotal belanja otomatis.

3. Kasir memproses transaksi dengan memilih metode pembayaran.

4. Setelah pembayaran terverifikasi, stok barang otomatis dikurangi dan struk kasir dicetak.

Proses Bisnis

🔄 Activity Diagram: Laporan

Alur owner dalam menarik statistik laporan keuangan.

Deskripsi Alur

1. Owner menavigasi ke menu Laporan pada sidebar admin.

2. Menentukan filter jangka waktu (tanggal awal dan akhir).

3. Controller meminta data transaksi ke database.

4. Sistem merender grafik profit dan data tabel transaksi lengkap dengan fitur ekspor PDF.

Interaksi Objek

🔁 Sequence Diagram: Login

Pesan sekuensial antarmuka selama proses login autentikasi pengguna.

Interaksi Sekuensial

Menggambarkan bagaimana objek `LoginForm` (Boundary), `AuthController` (Control), dan tabel `User` (Entity) berinteraksi untuk melakukan pengecekan hashing password sebelum memberikan otorisasi dashboard.

Interaksi Objek

🔁 Sequence Diagram: Penjualan

Pesan sekuensial antarmuka selama pembuatan transaksi dan pembayaran.

Interaksi Sekuensial

Visualisasi urutan pemanggilan API checkout belanja: penambahan item, kalkulasi sisa stok barang pada database, penyimpanan data pembayaran transaksi, hingga output respons cetak bukti struk fisik.

Interaksi Objek

🔁 Sequence Diagram: Laporan

Pesan sekuensial antarmuka penarikan data laporan oleh admin owner.

Interaksi Sekuensial

Menjelaskan pemrosesan filter tanggal dari admin panel menuju `ReportController`, query data agregat transaksi bulanan, hingga perangkaian data ke tabel visual dan unduh PDF.

Kolaborasi antar objek dalam merangkum omzet penjualan.

Status Entitas

⚙ Statechart Diagram: Login

Perubahan status internal sistem autentikasi kasir.

Status Transisi

Idle State: Menunggu input kredensial.

Validating State: Proses verifikasi password.

Authorized State: Akses diberikan ke menu dashboard.

Error Display State: Menampilkan pesan kesalahan jika verifikasi gagal.

Status Entitas

⚙ Statechart Diagram: Penjualan

Perubahan status transaksi belanja dalam sistem kasir.

Status Transisi

Empty State: Keranjang belanja kasir masih kosong.

Item Active: Barang ditambahkan ke dalam daftar keranjang.

Awaiting Payment: Menunggu pelanggan menyelesaikan transfer/pembayaran.

Paid State: Pembayaran lunas, transaksi disimpan permanen.

Status Entitas

⚙ Statechart Diagram: Laporan

Siklus hidup penyusunan rekap laporan keuangan.

Status Transisi

Unfiltered State: Halaman laporan default (tanpa filter tanggal).

Loading State: Proses query database sedang berjalan.

Rendered State: Statistik terisi dengan data omzet.

Exporting State: Pengunduhan dokumen cetak PDF.
